Error

에러는 예외와는 다르다. 프로세스를 실행 시킬 수 없는 상태이다. Java에서는 Error를 상속받은 클래스들이다.

예를 들어 StackOverflowError, OutOfMemoryError 들이 있다.

Exception

개발자의 실수, 환경에 의해 발생하는 예측 가능한 문제일 때 예외를 발생시킨다. 개발자가 핸들링 하도록 만들어 주는 것이다. Java에서는 Exception을 상속받은 클래스들이 해당된다.
Checked Exception RuntimeException의 상속을 받지 않은 Exception 클래스이다. Try-Catch/Throws를 강제한다. 만약 하지 않으면 컴파일이 되지 않는다. Spring에서는 기본적으로 트랜잭션 Rollback이 Checked Exception에 의해서는 실행되지 않는다. 설정에 의해 Rollback 예외를 지정할 수 있다.

FileNotFoundException, ClassNotFoundException 등이 있다.

Unchecked Exception RuntimeException을 상속받은 클래스이다. 예외처리를 강제하지 않는다. 실행 중에 발생할 수 있는 예외기 때문에 이를 모두 강제하게 되면 코드가 예외처리로 가득 할 것이다.

NullPointerException이 대표적이다.

적절한 예외와 예외처리 활용이 중요하다. 만약 모든 예외에 대해서 의미없는 try-catch로 묶어 버리면 문제가 생겼을 때 찾을 수 없게 만든다. 그렇다고 무조건 throws로 넘기게 된다면 코드가 복잡해지고 의미없는 예외조차 client 코드 단에서 처리해야 하는 일이 생길 것 이다.
