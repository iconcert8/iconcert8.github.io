HTTP Method

HTTP의 요청을 보낼 때 resource에 대한 원하는 행위를 규격화 한 것이다. 메소드 별로 특징이 있고 아래에서 알아보려고 한다.

GET
GET 메소드는 resource의 조회를 목적으로 한다. request body가 없다.

HEAD
HEAD 메소드는 GET과 동일하게 조회를 목적으로 하지만 본문을 응답받지는 않는다. 예를 들면, 파일 서버가 있을 때 header에 있는 파일 사이즈 정보만 알고 싶으면 파일 본문은 응답 받지 않고 header 정보만 조회할 수 있다.

POST
POST 메소드는 entity(데이터)의 제출을 목적으로 한다. header에 Content-Type을 추가해 어떤 타입의 데이터인지 명시해야한다.

PUT 메소드와 다른 점은 PUT 메소드는 멱등성이 있다. PUT 메소드를 여러번 실행 한다고 해서 데이터가 여러개 쌓이지 않는다. PUT 메소드는 데이터의 교체만 할 뿐이다. 반대로 POST는 데이터가 쌓이기 때문에 의도를 명확히 해야 한다.

PUT
PUT 메소드는 resource 생성/교체를 목적으로 한다. 위 POST 메소드에서 설명했던 것 처럼 PUT 메소드는 멱등성의 특징이 있다.

DELETE
DELETE 메소드는 resource의 삭제를 목적으로 한다.

CONNECT
CONNECT 메소드는 타켓 resource에 대해 서버와 터널을 설정한다. 중간 프록시 서버에게 타겟 서버와의 연결을 요청하면 프록시 서버에서 연결을 유지시켜 준다.

OPTIONS
OPTIONS 메소드는 타겟 resource와 통신 옵션에 대한 설명을 한다. 서버에게 허용된 통신옵션을 요청한다.

TRACE
TRACE 메소드는 타겟 resource 경로를 따라 loop-back(회귀) 테스트를 수행하여 디버깅에 도움을 준다. 최종 서버(Gateway, Proxy 들을 거치고 마지막 실사용 서버)에서 받은 request를 그대로 반환하여 준다. 이를 통해 요청이 예상대로 전달되었는지 파악이 용이하다.

PATCH
PATCH 메소드는 타겟 resource의 부분 변경을 목적으로 한다. CRUD에서 update와 유사하다. 그러나 일반적으로 HTTP는 CRUD와 다르며, 두 가지를 혼동해서는 안 된다.

PUT이 resource 전체를 표현하는 것과는 대조적으로 부분 표현이라는 차이가 있다.
