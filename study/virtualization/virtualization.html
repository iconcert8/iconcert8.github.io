Virtualization

물리적인 공간 내에서 논리적인 공간으로 분리하여 사용하는 환경
장점
통합성(Server Consolidation)

통합함으로써 물리적인 공간 비용 절약

독립성(Isolation)

독립적인 공간

효율성(Efficiency)

컴퓨팅 자원 사용 최대화

유연성(Flexibility)

서버 데이터 migration 용이

Hypervisor

VM(Virtual Machine)과 하드웨어간의 I/O 명령을 처리하는 인터페이스
Type 1

Bare-metal(빈깡통) 이라고도 하며, Host OS가 없고 게스트 OS들이 하드웨어 바로 위에 구동된다. Type 2보다 성능향상이 되지만 여러 하드웨어 드라이버 세팅이 필요하다.

전가상화

하드웨어 전체를 가상화 하는 방식. "DOM0"라는 관리용 가상머신을 통해 모든 명령이 번역되고 하드웨어에 전달된다.
결국, 하이퍼바이저가 관리 및 번역 업무를 모두 처리하는 것이다.

반가상화

게스트 OS의 번역업무를 게스트 OS 자체에서 처리한다. 하이퍼바이저는 게스트 OS 자원관리만 한다. "DOM0"를 거치지 않기 때문에 성능이 전가상화보다 좋다.
단, 게스트 OS에서는 하이퍼콜 인터페이스가 구현되어 있어야 한다. 게스트 OS의 코드 수정이 필요한 것이다.

Type 2

Host OS위에 하이퍼바이저를 실행한다. Type 1보다 성능이 떨어진다.
ex) Window OS에 Virtual Box를 설치하여 Linux를 사용

Hypervisor 단점

완전한 운영체제가 설치되기에 많은 메모리가 필요하다. CPU 성능을 확보(우선순위)하는 데에도 문제가 있다.


Container(컨테이너)

Host OS위에 컨테이너 소프트웨어를 설치하여 논리적인 공간인 컨테이너로 나누어 사용한다. OS 분리가 아닌 프로세스 격리를 한다. '모듈화된 프로그램 패키지'를 표방한다.

구동 환경이 달라짐에 따라 예상치 못한 오류가 발생할 확률이 높다. 필요 라이브러리, 바이너리, 기타 구성파일을 패키징하여 배포하여 오류를 최소화 한다.

컨테이너는 호스트 OS의 커널을 다른 컨테이너들과 공유한다. 이는 하이퍼바이저와 큰 차이점이다.
장점

- 시스템의 성능 부하가 적다. Host OS의 커널을 공유하기 때문에 프로세스 수준의 자원만 필요하다.
- 자원배분이 유연하다. CPU에 대한 사용량이나 사용자가 설정한 임계치에 따라 자동으로 확장/축소된다.
- 구동방식이 간단하다. 컨테이너 소프트웨어만 설치하면 쉽게 구동이 가능하다.

단점

타 OS구동이 안된다. 커널을 공유하기에 보안 취약성이 상대적으로 높다.
